gf 生成的主体业务项目结构

logic 层中的代码中

并没有对 dao 层进行一层封装，而是通过 dao 直接执行数据库的语句。不大优雅有点……

dao

使用 gf gen dao 生成的 dao 文件代码，可以自己添加填充。
dao/internal/name.go 中定义了具体的数据表的映射以及提供对应的方法进行操作.
dao/name.go 中可以封装自己的 dao 层方法，提供的方法可有对应的 logic 层调用，实现具体的业务逻辑。
类似于大家熟悉的 Model service controller 三层架构中的，Model 层。是实际进行数据库交互，CRUD 的地方。

service && logic

使用 gf gen service 根据对应 logic 生成对应的 service 代码，需要注意上面的注意点。
service是根据 logic层自动生成的一层抽象接口，可以根据不同的 logic 进行注册不同的 service 来保证上层 controller层调用不会发生影响。那么可以知道 logic是service的实现。

controller && api

对于这两个也是一起使用的 controller 和原始的 controller service model 三层中的架构是一样的，作用都是处理接收到的参数，以及调用 service 进行请求的处理。那么 api是什么呢？api中定义了，具体的传入参数的类型，以及对应的参数校验，以及具体请求返回的参数的类型。

do && entity

都是使用 gf gen dao 生成两个文件。具体的区别在于 service 调用 dao 的时候传入的 Data 是以 do中的模型为数据模型，在 dao 调用 scan 以 entidy 为模型。
